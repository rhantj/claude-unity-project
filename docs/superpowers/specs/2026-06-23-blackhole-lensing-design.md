# 블랙홀 비주얼 시뮬레이터 — 설계 (2단계: 레이마칭 렌즈)

- 작성일: 2026-06-23
- 엔진: Unity 6 (6000.5.0f1), URP 17.5.0
- 범위: **2단계 = 비주얼**. 1단계(동역학: `BlackHole`/`GravityField`/`GravityIntegrator`/`ParticleField`/`ThrowableBody`/`SimController`)는 변경하지 않고 그대로 사용한다.

## 목표

블랙홀 주변 화면을 실시간 레이마칭으로 휘게 그려서(중력 렌즈), 그림자(이벤트 호라이즌), 발광하는 강착원반, 별이 흐르는 어두운 우주 배경을 갖춘 인터스텔라풍 비주얼을 만든다. 1단계의 입자/던진 물체는 그 위에 일반 렌더로 합성한다.

## 결정 사항

| 항목 | 결정 |
|---|---|
| 렌즈 방식 | **실시간 레이마칭** (슈바르츠실트 측지선 근사). 화면 전체를 매 픽셀 마칭 |
| 통합 방식 | URP `ScriptableRendererFeature` + 풀스크린 Blit 패스 |
| 배경 | 절차적 별(해시 노이즈 기반) — 외부 큐브맵/텍스처 에셋 불필요 |
| 강착원반 | 셰이더 내 해석적 모델(적도면 환형, 온도 그라데이션 + 도플러 비대칭) |
| 합성 순서 | 레이마칭 패스(배경+렌즈+그림자+원반) → 그 위에 씬의 실제 오브젝트(입자, 던진 물체, 궤적)를 일반 렌더 |
| 1단계 입자에 대한 렌즈 영향 | **없음(범위 밖)**. 입자/물체 자체는 휘어 보이지 않음 — 셰이더의 해석적 원반이 "휘어진 원반" 그림을 전담 |
| Bloom/톤매핑 | URP Volume Profile (Bloom + Tonemapping ACES) |
| 신규 패키지 | 추가하지 않음 (URP 기본 기능만 사용) |

## 왜 이 구성인가 (대안 비교)

1. **(채택) 레이마칭 패스가 배경을 그리고, 1단계 객체는 그 위에 별도 렌더**
   - 장점: 1단계 물리/코드를 전혀 건드리지 않음. 디버깅 시 "배경만" 따로 끄고 볼 수 있어 단계적 검증이 쉬움.
   - 단점: 입자/던진 물체 자체는 광학적으로 휘지 않음 (다만 화면 대부분을 차지하는 배경·원반의 왜곡이 시각적 효과를 지배하므로 체감상 자연스러움).
2. (기각) 씬 전체를 깊이 포함해서 레이마칭으로 재합성(입자까지 광학적으로 왜곡)
   - 인터스텔라 수준에는 더 가깝지만 깊이 기반 합성이 필요해 구현/디버깅 난이도가 크게 증가. YAGNI — 1단계 객체는 적고 화면 중심에서 멀리 있을 때가 많아 효과 대비 비용이 낮음.
3. (기각) 버텍스 셰이더로 배경 메쉬(스카이스피어)를 직접 왜곡
   - 레이마칭 없이 가볍지만 광자구·아인슈타인 링 같은 강한 곡률 효과를 표현할 수 없음. 애초에 사용자가 고른 "실시간 광선 레이마칭"의 핵심을 포기하는 셈.

## 레이마칭 모델

- **광선 가속도(근사)**: 각 마칭 스텝에서 광선 위치 `p`, 방향 `d`에 대해 `BlackHole`과 동일한 `GravityField.AccelerationAt(p)`를 광자 경로 휨에 맞춘 계수로 사용한다. 정확한 측지선 방정식 대신, 우리 1단계 중력 코어와 동일한 형태의 `a = -k · r / (r² + ε²)^(3/2)` 식으로 방향을 굴절시키는 **근사 광선 추적**을 쓴다. 물리적으로 완전한 GR은 아니지만 시각적으로 광자구·링 효과를 내기에 충분하고, 우리 코어 수식과 일관성이 있어 설명하기 쉽다.
- **스텝 적분**: 고정 스텝 수(인스펙터 노출, 기본값으로 시작해 성능 보고 조정) 동안 `d += a(p) * stepSize; p += d.normalized * stepSize`를 반복.
- **종료 조건**: (a) `|p - blackHoleCenter| <= eventHorizonRadius` → 그림자(검정) 반환, (b) 강착원반 환형(적도면, 내경~외경)과의 교차 → 원반 색 반환 후 마칭 종료, (c) 스텝 한도 도달 또는 충분히 멀어짐 → 별 배경 샘플.
- **별 배경**: 최종 광선 방향 `d`를 구면 좌표로 해시하여 절차적 별점(밀도+밝기 노이즈) 생성. 텍스처 에셋 불필요.
- **강착원반 셰이딩**: 반경에 따른 온도 매핑(안쪽 청백색 ~ 바깥 주황/적색), 광원-카메라 상대 방향에 따른 도플러 비대칭(접근측 가산 밝기), 렌즈로 인해 원반의 "뒷면"도 위/아래로 보이는 효과는 마칭이 원반 평면과 여러 번 교차할 수 있게 해 자연 발생시킨다(첫 교차만 쓰지 않고 마칭 도중 평면 통과를 모두 검사).

## 컴포넌트

| 파일 | 역할 | 의존 |
|---|---|---|
| `BlackBodyColor.cs` (Runtime) | 온도(K) → RGB 근사 색 변환 순수 함수. 셰이더의 강착원반 색 램프와 동일 상수를 공유하는 기준점 | 없음 |
| `BlackHoleLensController.cs` (Runtime) | 매 프레임 `BlackHole`의 월드 위치/`EventHorizonRadius`와 원반 파라미터(내경/외경/온도 범위/마칭 스텝)를 전역 셰이더 프로퍼티로 push | `BlackHole` |
| `BlackHoleLens.shader` (Shaders) | 풀스크린 HLSL: 카메라 광선 생성 → 레이마칭 → 그림자/원반/별 출력 | `BlackBodyColor`와 동일한 색 상수(셰이더에 하드코딩, 주석으로 C# 출처 명시) |
| `BlackHoleLensFeature.cs` (Runtime/Editor 경계 없음, Renderer Feature는 런타임 어셈블리에 둠) | `ScriptableRendererFeature` + `ScriptableRenderPass`: URP Color 타겟에 풀스크린 Blit로 셰이더 결과를 그려 넣음(카메라 색 버퍼 위, 오파크 지오메트리 패스 이전 — 즉 "배경"으로 취급되어 1단계 오브젝트가 그 위에 자연스럽게 그려짐) | `BlackHoleLens.shader` |
| Editor 헬퍼 (`BlackHoleSimEditorTools.cs`에 메서드 추가) | 현재 활성 URP `UniversalRendererData` 에셋의 `rendererFeatures` 리스트에 `BlackHoleLensFeature`를 등록(중복 등록 방지) | 기존 Editor asmdef 재사용 |
| URP Volume Profile (씬 에셋) | Bloom(threshold/intensity 조정) + Tonemapping(ACES) | - |

## 데이터 흐름

`BlackHoleLensController.Update` → `Shader.SetGlobalVector/Float`로 블랙홀 위치, `eventHorizonRadius`, 원반 내·외경, 온도 범위, 마칭 스텝 수, 스텝 크기를 전역 프로퍼티에 기록 → `BlackHoleLensFeature`의 렌더 패스가 매 프레임 풀스크린 Blit으로 `BlackHoleLens.shader` 실행 → 결과가 카메라 컬러 타겟에 "배경"으로 그려짐 → 이후 URP의 일반 오파크 패스가 1단계 씬 오브젝트(블랙홀 placeholder 구체는 더 이상 시각적 블랙홀 역할을 하지 않으므로 비활성화하거나 그대로 두어 셰이더 그림자와 겹쳐도 무방, 입자, 던진 물체, 트레일)를 그 위에 그림 → URP Volume의 Bloom/Tonemapping이 최종 후처리.

## 씬 변경 (MCP로 적용)

- 기존 `BlackHole` 구체(검은 Unlit 머티리얼)는 셰이더 그림자와 겹치므로 **비활성화**하거나 셰이더 그림자 반경과 맞춰 작게 유지한다(완전 비활성화를 기본으로 하고, 필요 시 인스펙터에서 다시 활성화 가능하게 구조는 유지).
- 새 GameObject `BlackHoleLensController` + 컴포넌트, `blackHole` 참조 와이어링.
- URP Renderer 에셋에 `BlackHoleLensFeature` 추가(Editor 헬퍼).
- Global Volume 추가 + Bloom/Tonemapping 활성화.

## 에지 케이스

- 카메라가 이벤트 호라이즌 안쪽으로 줌인되는 경우: 마칭 시작점이 이미 그림자 영역이면 화면 전체가 즉시 검정 — 의도된 동작(설계상 카메라 줌은 `SimController`의 `zoomSpeed`로 제한되어 있어 호라이즌 통과는 일반 사용 흐름에서 드묾, 별도 클램프는 추가하지 않음).
- 마칭이 스텝 한도 내에 별/그림자/원반 어느 것도 만나지 못하는 경우: 기본 별 배경 색(아주 약한 별 밀도) 반환 — 검정 화면 방지.
- 강착원반 평면을 거의 정면(탑뷰)으로 볼 때: 환형이 가늘게 보이는 것은 물리적으로 올바른 동작이므로 별도 보정 없음.

## 테스트 전략

- `BlackBodyColor`: EditMode 단위 테스트로 대표 온도(저온=주황, 고온=청백) 입력에 대해 RGB 채널 우세 관계를 검증. 1단계 테스트 스위트와 같은 `BlackHoleSim.Tests` asmdef에 추가.
- 셰이더/레이마칭 자체는 단위 테스트 불가 영역이므로 **단계적 스크린샷 검증**으로 대체:
  1. Renderer Feature 배선만 — 단색 출력으로 패스가 실제로 실행되는지 확인
  2. 별 배경만 — 절차적 별이 보이는지
  3. 렌즈 굴절 — 배경 별의 왜곡(아인슈타인 링 근처 패턴)
  4. 그림자 — 중심 검정 원
  5. 강착원반 — 온도 그라데이션 + 비대칭 밝기
  6. Bloom 적용 후 최종

## 튜닝 가능 파라미터 (인스펙터)

- 마칭 스텝 수, 스텝 크기(고정 vs 적응형은 고정으로 시작)
- 굴절 계수 `k`(렌즈 세기)
- 강착원반 내경/외경, 온도 범위(K), 도플러 비대칭 강도
- 별 밀도/밝기
- Bloom intensity/threshold

## 범위 밖 (YAGNI)

- 회전 블랙홀(커 메트릭), 다중 블랙홀
- 1단계 입자/던진 물체 자체의 광학적 왜곡(깊이 기반 합성)
- 정확한 GR 측지선 적분(상대론적 광선 방정식 전체 구현)
- 외부 큐브맵/별자리 텍스처 임포트
